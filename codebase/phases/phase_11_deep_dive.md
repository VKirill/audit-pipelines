# PHASE 11 — DEEP DIVE (forensic-grade анализ критичных зон)

**Цель:** Forensic-grade анализ выбранных критичных зон. Превратить "first-pass risk scan" в "готовый без оговорок" аудит для зон, которые того стоят.

**Когда запускать (v3 правило):**

| Сценарий | Phase 11 |
|----------|----------|
| ≥ 1 critical finding в `findings.jsonl` | **ОБЯЗАТЕЛЬНО** — `finalize.sh` падает без `audit/11_*.md`. Зона deep-dive выбирается агентом без обсуждения, по принципу: первая зона = `subcategory` первого critical, вторая = крупнейший concentration кластера findings. |
| Только high/medium findings | **Опционально**. Агент предлагает 4–6 кандидатов, пользователь выбирает 2–4. |
| Mostly low/info | Можно пропустить. |

В v2 фаза была опциональной всегда. В v3 при наличии critical — это hard requirement, потому что critical-finding должен быть подтверждён forensic-grade перед тем, как ROADMAP направит команду на «срочную ротацию» / «срочный hotfix».

**Источники:**
- Feathers, *Working Effectively with Legacy Code* — characterization tests, seam points.
- Fowler, *Refactoring* 2e — механика безопасных изменений.
- Nygard, *Release It!* — operational review.

## 2. Процесс выбора зон

### 2.1. Кандидаты-зоны

Смотрим на:
1. Зоны, попавшие в **Phase 0** (Critical) ROADMAP — нужны для подтверждения риска перед срочным фиксом.
2. Зоны, попавшие в **Phase 2 Quarterly Epics** с `effort: L/XL` — перед большой инвестицией нужна уверенность.
3. Центральные символы с `confidence: medium` — где в первой фазе не хватало глубины.
4. Зоны с большой концентрацией findings (> 5 findings в одном модуле/папке).

### 2.2. Обсуждение с пользователем

Агент **не выбирает сам**. Агент предлагает 4–6 кандидатов с обоснованием, пользователь выбирает 2–4 для deep-dive. Формат предложения:

```
Предлагаю следующие зоны для второго прохода (выбери 2-4):

1. **apps/bot/src/shared/lib/image-generation/**
   - Зачем: здесь сосредоточены 3 performance findings (F-0009, F-0011, F-0014) и 1 resilience finding (F-0008), но чтение тел функций было эвристическим.
   - Что получим: подтверждение/опровержение sync-I/O race и N+1 в queue path; seam points для рефакторинга.
   - Оценка времени: ~30 минут.

2. **apps/web/server/api/admin/** и auth/session boundary
   - Зачем: admin-доступ имеет shared password (F-0005, high), но full route-by-route не завершён.
   - Что получим: полная таблица ACL по всем admin эндпоинтам, конкретные CVE-like findings если они есть.
   - Оценка времени: ~45 минут.

3. ...
```

## 3. Процесс deep-dive для выбранной зоны

Для каждой выбранной пользователем зоны:

### 3.1. Подготовка

- [ ] Прочитай ВСЕ связанные findings из `findings.jsonl` по location, содержащему эту зону.
- [ ] Составь список файлов в зоне (`list_dir --recursive`).
- [ ] Определи границу зоны точно: куда она заходит, куда нет.

### 3.2. Полный проход по файлам

**Это не семпл. Это все файлы зоны.**

Для **каждого** файла в зоне:

- [ ] `get_symbols_overview` (или `head -50` + ручной скан).
- [ ] Для **каждой** функции/класса в файле:
  - [ ] `find_symbol include_body=true` / ручное чтение тела.
  - [ ] Проверь по всем чеклистам фаз 04, 05, 06, 09 (те что применимы).
  - [ ] Зафиксируй в журнале наблюдения — файл, символ, наблюдение (даже мелкое).
- [ ] `find_referencing_symbols` / grep на каждый символ — кто вызывает.
- [ ] Для каждой функции, которая делает I/O, auth, crypto или обращение к БД — **дополнительная** проверка на 3 вопроса:
  1. Что если вход невалиден?
  2. Что если внешний сервис не отвечает?
  3. Что если два вызова происходят одновременно?

### 3.3. Seam analysis (Feathers)

Для этой зоны определи:
- [ ] Где **уже есть** естественные seams (интерфейсы, DI-точки, границы модулей)?
- [ ] Где seams **нужно создать** для безопасного тестирования / рефакторинга?
- [ ] Для каждого кандидата на seam — конкретный рефакторинг-шаг с минимальным риском (Sprout Method / Sprout Class / Wrap Method — по Feathers).

### 3.4. Concurrency / transaction trace

Если зона включает hot path:
- [ ] Трассируй один полный вызов через зону — от entry до выхода.
- [ ] Отметь: где shared state, где transactions, где external I/O, где timeouts.
- [ ] Оцени: что произойдёт при 10 одновременных вызовах? При таймауте на внешнем сервисе в середине? При retry?

### 3.5. Дополнительные findings

Все новые находки этой фазы — с id `F-11NNN` (сквозная нумерация продолжается, префикс 11 для deep-dive).

Confidence для deep-dive findings обычно **выше**, чем в основном проходе, потому что опирается на полное чтение. Но правила §3.3 оркестратора всё равно действуют.

### 3.6. Обновление существующих findings

- [ ] Findings, которые подтвердились при глубокой проверке → повысить confidence на ступень (и в `evidence` дополнить цитатой из тела функции).
- [ ] Findings, которые **опровергнуты** (оказались false positive после чтения) → пометить `status: invalidated`, в `related_findings` отметить «опровергнуто в deep-dive». **Не удаляй** — это часть истории.
- [ ] Findings, которые нужно переклассифицировать (severity изменился) → обновить с пометкой `updated_in: phase_11`.

## 4. Артефакт — `audit/11_deep_dive.md`

### Обязательные разделы

1. **Цель фазы** (какие зоны выбраны и почему)
2. **Зоны, проанализированные в этом проходе** (список с датами/временем)
3. **Для каждой зоны — отдельный подраздел:**
   - **Границы зоны** — какие файлы включены.
   - **Полный inventory функций/классов** — таблица.
   - **Concurrency / transaction trace** (если применимо) — последовательность вызовов.
   - **Seam points** — таблица существующих и создаваемых.
   - **Новые findings** — список id + краткое описание.
   - **Обновлённые findings** — список id + что изменилось.
   - **Опровергнутые findings** — список id + почему.
   - **Рекомендации по рефакторингу/фиксам** — с явным шагом за шагом.
4. **Общие выводы** — что этот проход изменил в общей картине?
5. **Обновления ROADMAP** — нужно ли добавить/убрать/переприоритизировать эпики после deep-dive? Список конкретных изменений.
6. **Следующие шаги** — должно ли быть ещё один deep-dive по другим зонам?

## 5. Обязательные evidence файлы

Для каждой зоны — отдельная подпапка в `audit/evidence/11_deep_dive/<zone_slug>/`:

- `inventory.md` — полный список файлов и символов.
- `trace_<scenario>.md` — трассировка 1–2 основных сценариев (для hot path зон).
- `seams.md` — анализ seam points.
- `new_findings.md` — таблица новых finding'ов из этой зоны.

## 6. Обновление ROADMAP

После deep-dive **обязательно** вернись к `audit/ROADMAP.md` и обнови:

- [ ] Executive summary — если общее состояние зоны изменилось.
- [ ] Phase 0/1/2 — если какие-то задачи оказались более/менее срочными.
- [ ] Метрики успеха — если появились новые measurable targets.
- [ ] Dependency DAG — если deep-dive обнаружил, что один эпик должен предшествовать другому.

## 9. Когда deep-dive НЕ нужен (только если 0 critical)

Если в `findings.jsonl` нет critical — фаза опциональна. Пропусти, если:
- Основные findings имеют `confidence: high` с цитатами строк.
- ROADMAP ясный, команда может начинать работу.
- Проект небольшой (XS/S), основной проход уже покрыл всё глубоко.
- Phase 0 содержит только тривиальные задачи (ротация ключей, включение CI).

Deep-dive стоит запускать, если:
- В Phase 2 есть эпик с `effort: L/XL` и команда готова его начинать.
- Критичный путь имел `confidence: medium` из-за деградации инструментов.
- В одном модуле сконцентрировано > 5 findings — стоит убедиться, что они действительно разные, а не дубликаты одного корня.

**Если в `findings.jsonl` есть ≥ 1 critical — фаза обязательна.** `finalize.sh` падает без `audit/11_*.md` в этом случае.

## 10. Exit gate

```bash
bash audit_pipeline/scripts/validate_phase.sh 11
```

Проверки:
- Отчёт `audit/11_*.md` существует и ≥ 150 строк.
- 7 разделов присутствуют.
- Если есть **новые** findings — confidence_rationale + (для critical) exploit_proof обязательны.
- Цитированные file:lines резолвятся через `check_evidence_citations.py`.
