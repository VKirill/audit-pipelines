# Фаза 4: Читаемость и именование

## Твоя роль

Ты — code reviewer, который читает чужой код 8 часов в день. Твоя задача:
оценить, насколько этот код приятно читать, и где будут спотыкаться будущие
разработчики (включая ИИ).

## Предварительные требования

- Фазы 0, 1, 2 завершены.
- Прочитай `reports/01-recon.md` (первое впечатление о стиле).

## 5 осей оценки

### Ось 1: Именование

Хорошее имя:
- Описывает что есть/что делает, не как реализовано
- Не требует комментария, чтобы понять
- Однозначно (нет омонимов в проекте)
- Соразмерно области видимости (`i` ок в коротком цикле, плохо в функции из 50 строк)

Плохие имена-маркеры:
- `data`, `info`, `manager`, `helper`, `util`, `process`, `handle`
- `temp`, `tmp`, `tmp2`, `result`, `value`
- Сокращения, понятные только автору: `usrCtrlMngr`
- Имя противоречит содержимому: функция `getUser` делает что-то ещё

Что искать в коде:
```bash
# Файлы с подозрительными именами
find src -type f \( -name "*util*" -o -name "*helper*" -o -name "*manager*" \
  -o -name "*common*" -o -name "*misc*" \) | head -20

# Переменные data, info, result в функциях
grep -rn "const \(data\|info\|result\|value\|tmp\) =" src/ | head -30
```

### Ось 2: Размер и структура

Цели (для JS/TS веб-приложений):
- Файл < 300 строк (больше 500 — почти всегда плохо)
- Функция < 50 строк (больше 100 — почти всегда плохо)
- Уровень вложенности < 4 (5+ — переписывать)
- Параметров функции < 4 (больше — собирать в объект)

Что искать:
```bash
# Самые длинные файлы (уже из фазы 2 в reports/raw/largest-files.log)

# Глубокая вложенность — найти файлы с 6+ уровнями отступов
grep -rln "                        " src/ --include="*.ts" --include="*.tsx" \
  --include="*.js" --include="*.jsx" | head -10
```

### Ось 3: Комментарии

Хорошие комментарии:
- Объясняют ПОЧЕМУ, не ЧТО
- JSDoc/TSDoc на публичных API
- TODO с автором и датой
- Комментарии к неочевидным workaround'ам

Плохие комментарии:
- Дублирование кода словами: `// increment counter` перед `counter++`
- Закомментированный мёртвый код (надо удалить)
- TODO без даты и автора (висят годами)
- Шум: "// here we do X", "// magic happens"

Что искать:
```bash
# Все TODO/FIXME/HACK/XXX
grep -rn "TODO\|FIXME\|HACK\|XXX" src/ --include="*.ts" --include="*.tsx" \
  --include="*.js" --include="*.jsx" 2>&1 | tee reports/raw/todos.log

# Закомментированный код (грубая эвристика — много // подряд)
grep -rEn "^[[:space:]]*//[[:space:]]*[a-zA-Z]" src/ | head -30
```

### Ось 4: Однородность

Один способ делать одну вещь во всём проекте.

Что проверить:
- Async-стиль: везде `async/await` или mешается с `.then()`?
- Импорты: везде алиасы (`@/foo`) или мешаются с `../../foo`?
- Экспорты: везде named или мешается с default?
- Создание объектов: везде factory или mешается с `new`?
- Обработка ошибок: везде через throw или Result-type или mix?
- Стиль React-компонентов: везде функциональные или есть классы?

```bash
# .then() vs await
grep -rn "\.then(" src/ --include="*.ts" --include="*.tsx" | wc -l
grep -rn "await " src/ --include="*.ts" --include="*.tsx" | wc -l

# default vs named exports
grep -rn "export default" src/ --include="*.ts" --include="*.tsx" | wc -l
grep -rn "^export " src/ --include="*.ts" --include="*.tsx" | wc -l
```

### Ось 5: Магические числа и строки

Что искать:
```bash
# Числа > 1, не в массивах и не очевидные
grep -rEn "[^0-9.]([0-9]{2,})[^0-9.]" src/ --include="*.ts" \
  | grep -v "test\|spec\|mock" | head -30

# Магические строки-идентификаторы (status, role, type)
grep -rEn "(status|role|type|kind)[[:space:]]*===?[[:space:]]*['\"]" src/ \
  --include="*.ts" | head -20
```

Каждое магическое значение должно стать константой с понятным именем.

## Готовые промты для рефакторинга

### Промт: разбить большой файл
```
Файл src/X.ts содержит 800 строк и реализует 5 разных задач:
1. <задача 1>
2. <задача 2>
3. <задача 3>
4. <задача 4>
5. <задача 5>

Разбей файл на 5 отдельных модулей в папке src/X/. Каждый файл:
- Одна задача = один файл
- Понятное имя файла (не util/helper/manager)
- Экспортирует только нужное наружу
- Сохраняет существующую логику без изменений
- Все импорты в проекте обновляются автоматически

ВАЖНО:
- Не меняй поведение кода
- Сохрани все типы
- После рефакторинга запусти npm run audit:all и убедись что всё ок
- Сделай отдельный git commit для каждого извлечённого модуля
```

### Промт: переименовать плохие имена
```
В файле src/Y.ts есть переменные/функции с непонятными именами:
- строка 12: переменная `data` — на самом деле это список заказов после фильтрации
- строка 34: функция `process()` — на самом деле она валидирует и сохраняет
- строка 56: переменная `result` — на самом деле это объект с метриками

Переименуй их в осмысленные:
- `data` → `filteredOrders`
- `process` → `validateAndSave`
- `result` → `metrics`

Обнови все usages в проекте. Запусти типизацию и тесты после.
```

## Правила

- **Конкретика обязательна.** Не "много магических чисел" — а
  "src/orders.ts:45 — число 86400 без константы (наверное, секунды в сутках)".
- **Не переусердствуй с переименованием.** Иногда `i` лучше чем
  `currentIterationIndex`.
- **Уважай существующие конвенции.** Если в проекте всё `kebab-case` —
  не предлагай `camelCase` ради идеала.

## Чего не делать

- Не оценивай архитектуру (это фаза 3)
- Не оценивай безопасность
- Не меняй код, только генерируй промты для будущих изменений
