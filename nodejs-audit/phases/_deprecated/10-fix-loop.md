# Фаза 10: Цикл исправлений

## Когда использовать

После того, как у тебя есть `FINAL-REPORT.md` из фазы 9, и ты готов
исправлять. Эта фаза — не один прогон, а цикл, который запускается
многократно.

## Как использовать

В Claude Code:

```bash
claude "Прочитай nodejs-audit/10-fix-loop.md и выполни цикл #N"
```

где N — номер промта из FINAL-REPORT.md, который ты хочешь применить.

## Твоя роль

Ты — инженер, который применяет конкретное исправление. Минимально, точно,
безопасно. Без расширения скоупа.

## Алгоритм одного цикла

### Шаг 1: Принять задачу

Открой `reports/FINAL-REPORT.md`. Найди промт #N. Прочитай его полностью.

Если в задаче что-то непонятно — задай 1-2 уточняющих вопроса владельцу
и остановись.

### Шаг 2: Snapshot до изменений

```bash
# Текущее состояние тестов и линтера
npm run audit:all 2>&1 | tee reports/raw/before-fix-N.log
git status
git log -1 --oneline
```

Создай файл `reports/fix-N/BEFORE.md` с этими данными.

### Шаг 3: Создать ветку

```bash
git checkout -b fix/audit-N-<краткое-описание>
```

### Шаг 4: Применить изменения

Делай **минимальные** изменения. Один логический шаг = один коммит.

Правила:
- Не делай рефакторинг "за компанию"
- Не меняй стиль соседнего кода
- Не добавляй новые зависимости без уведомления владельца
- Не удаляй существующие тесты без замены

### Шаг 5: Проверка после каждого подшага

После каждого коммита:

```bash
# Минимум — типы и тесты
npm run audit:types
npm run audit:test

# Если меняется поведение — полный прогон
npm run audit:all
```

Если что-то сломалось — откатись к последнему рабочему состоянию:
```bash
git reset --hard HEAD~1
```

И попробуй меньшим шагом.

### Шаг 6: Snapshot после изменений

Создай `reports/fix-N/AFTER.md`:

```markdown
# Fix #N: <название>

## До
- ESLint errors: X
- ESLint warnings: Y
- TypeScript errors: Z
- Tests: A passed, B failed
- ...

## После
- ESLint errors: X' (изменение: ...)
- ESLint warnings: Y'
- TypeScript errors: Z'
- Tests: A' passed, B' failed
- ...

## Что сделано
- [список коммитов]

## Что НЕ сделано (хотя могло бы)
- [список вещей, которые тоже стоит починить, но не входили в скоуп]
- Эти задачи — кандидаты на следующие циклы.

## Риски
- [что может сломаться, что нужно проверить вручную]

## Готовность к merge
- [ ] Все тесты проходят
- [ ] Линтер чист
- [ ] Типы чистые
- [ ] Code review (если есть кому)
```

### Шаг 7: PR / Merge

Если есть GitHub:
```bash
gh pr create --title "fix(audit): #N <описание>" --body "$(cat reports/fix-N/AFTER.md)"
```

Если нет — попроси владельца смержить локально:
```bash
git checkout audit-2026
git merge fix/audit-N-...
```

### Шаг 8: Закрыть цикл

Обнови `reports/FIXES-LOG.md`:

```markdown
# Лог исправлений

| # | Дата | Промт | Статус | Коммиты |
|---|------|-------|--------|---------|
| 1 | <date> | <название> | ✅ merged | abc123, def456 |
| 2 | <date> | ... | 🚧 in progress | ... |
```

## Правила безопасности

### НЕ делай

- ❌ Не применяй несколько промтов в одной сессии — потеряешь контекст
- ❌ Не объединяй "пока я тут, заодно..." с основной задачей
- ❌ Не игнорируй упавшие тесты со словами "потом починю"
- ❌ Не пуш в master/main напрямую
- ❌ Не удаляй файлы из reports/

### Делай

- ✅ Один промт = одна сессия = одна ветка
- ✅ Маленькие коммиты с понятными сообщениями
- ✅ Тесты после каждого коммита
- ✅ Если застрял — остановись и спроси владельца
- ✅ Документируй что сделал в reports/fix-N/AFTER.md

## Сложные случаи

### Промт требует изменений в N файлах

Разбей на под-задачи. Каждый файл — отдельный коммит. После каждого
файла — тесты.

### Тесты падают, и непонятно почему

```bash
# Откатись к last green
git reset --hard <last-green-commit>

# Применяй меньшими шагами
# Если на каждом шаге пытаешься воспроизвести проблему
```

Если не получается за 30 минут — остановись, опиши что пробовал в
`reports/fix-N/STUCK.md`, попроси владельца помощи.

### Промт требует решения, которого нет в проекте

Например, промт говорит "используй pino", но в проекте нет logger вообще.

В таком случае:
1. Не угадывай. Остановись.
2. Опиши ситуацию владельцу: "промт предполагает X, но в проекте есть Y.
   Варианты: A, B, C. Что выбрать?"
3. Жди ответ.

### Промт устарел (после других исправлений)

Если предыдущие циклы изменили проект, и промт N больше не релевантен —
не пытайся его применить как есть. Опиши ситуацию владельцу и предложи
обновлённую версию промта.

## После цикла: что дальше

Когда цикл закрыт:

1. Если есть ещё неприменённые промты — переходи к следующему по приоритету
2. Если все P0 и P1 закрыты — запусти **повторный частичный аудит**
   (фазы 02 и 09) чтобы измерить прогресс
3. Если прошёл месяц — запусти **полный аудит** (все фазы) заново
