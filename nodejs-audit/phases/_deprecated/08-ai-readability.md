# Фаза 8: AI-readability

## Твоя роль

Ты — ИИ-ассистент, который завтра будет дорабатывать этот код. Это самая
важная фаза для владельца, потому что вся будущая разработка идёт через ИИ.

Оцени: насколько легко тебе (или другой ИИ-модели) будет понять, изменить,
расширить этот код, не сломав ничего.

## Предварительные требования

- Все предыдущие фазы (0-7) завершены.
- Прочитай все отчёты в `reports/`.

## 6 осей оценки

### Ось 1: Понимание проекта за 5 минут

Если бы я открыл этот репозиторий впервые — за 5 минут я понял бы:
- Что это за приложение?
- Как его запустить?
- Где лежит главная бизнес-логика?
- Какие конвенции в этом проекте?

Что проверить:
- Есть ли AGENTS.md / CLAUDE.md в корне?
- Есть ли README.md и он осмысленный (не просто "npm start")?
- Описаны ли конвенции?
- Есть ли .env.example?

### Ось 2: Навигация

- Понятны ли имена папок и файлов?
- Легко ли найти где живёт фича X?
- Не разбросана ли логика одной фичи по 10 разным местам?

Тест: возьми произвольную пользовательскую фичу (например, "регистрация
нового пользователя") и попробуй найти все её части. Если приходится
прыгать по 8+ файлам в разных папках — низкая оценка.

### Ось 3: Однородность

ИИ работает лучше, когда есть один способ делать одну вещь. Тогда он
по примеру может добавить новый модуль.

Что проверить:
- Если я хочу добавить новый эндпоинт — есть ли явный шаблон?
- Если я хочу добавить новую таблицу в БД — есть ли явный путь?
- Если я хочу добавить новый компонент — есть ли пример?

Из фазы 4 уже есть данные про однородность (async/await, exports и т.д.).

### Ось 4: Контекст в коде

ИИ читает код небольшими кусками. Если контекст не в самом куске — ИИ
галлюцинирует.

Что проверить:
- JSDoc/TSDoc на публичных функциях?
- Типы вместо вывода типов в публичных API (явный возвращаемый тип)?
- Магические числа объяснены константами?
- Краткие комментарии для неочевидных решений?

```bash
# Сколько публичных функций имеют JSDoc?
grep -rEn "^/\\*\\*$" src/ --include="*.ts" | wc -l
grep -rEn "^export (async )?function" src/ --include="*.ts" | wc -l
# Отношение даёт примерное покрытие документацией
```

### Ось 5: Тесты как документация

Хорошие тесты — лучшая документация для ИИ.

Что проверить:
- Имена тестов описательные? ("должен X когда Y")
- Покрывают ли тесты основные сценарии?
- Можно ли по тестам понять API, не читая код?

```bash
# Имена тестов
grep -rEn "(it|test)\\(['\"]" src/ tests/ --include="*.test.*" --include="*.spec.*" | head -30
```

### Ось 6: Безопасность доработок

ИИ может сломать что-то. Какие есть защитные механизмы?

- Есть ли тесты, которые поймают регрессию?
- Включена ли строгая типизация?
- Запускается ли линтер на коммите (husky/lint-staged)?
- Есть ли CI, который не пропустит сломанный код?

```bash
# Husky
ls .husky/ 2>/dev/null

# CI
ls .github/workflows/ 2>/dev/null

# Strict TypeScript
grep -E "\"strict\":" tsconfig.json 2>/dev/null
```

## Готовые промты

### Промт: создать полный AGENTS.md
```
Прочитай весь проект (структура, package.json, основные файлы)
и создай AGENTS.md в корне со следующими секциями:

1. Что это за проект (3 предложения)
2. Стек технологий (точные версии)
3. Структура папок (с описанием что в каждой)
4. Как запустить локально (пошагово)
5. Как тестировать
6. Конвенции:
   - Именование файлов
   - Именование функций/переменных
   - Стиль импортов
   - Архитектурные правила (что в каком слое)
7. Что НЕЛЬЗЯ делать в этом проекте (anti-patterns)
8. Шаблоны:
   - Как добавить новый эндпоинт (пример из проекта)
   - Как добавить новую таблицу (пример)
   - Как добавить новый компонент (пример если есть фронт)
9. Контакты владельца / где задавать вопросы

Используй существующие файлы как источник правды. Не выдумывай конвенции —
выводи из того что уже есть в коде.
```

### Промт: добавить JSDoc на публичные функции
```
Добавь TSDoc-комментарии на все публичные функции в папке src/services/.

Формат:
/**
 * Краткое описание (одно предложение).
 *
 * Подробности если нужно (что делает, какие side effects).
 *
 * @param X - описание
 * @param Y - описание
 * @returns описание
 * @throws SomeError если ...
 */

ВАЖНО:
- Не меняй сами функции
- Если непонятно что делает функция — пропусти и пометь как [JSDOC-TODO]
- Один файл = один git commit
- После каждого файла — запусти npm run audit:types
```

### Промт: включить strict mode
```
Включи в tsconfig.json:
- strict: true
- noUncheckedIndexedAccess: true
- exactOptionalPropertyTypes: true

Запусти tsc --noEmit, посчитай количество ошибок.

Если ошибок < 50 — исправь все.
Если 50-200 — исправь по слою (services, controllers, utils, ...).
Если > 200 — сначала добавь "skipLibCheck: true" и используй // @ts-expect-error
с TODO для самых сложных мест, чтобы быстро включить strict, потом
постепенно убирай ts-expect-error.

Важно: одно изменение = один коммит. После каждого — запусти тесты.
```

## Правила

- Это самая важная фаза для владельца — не торопись.
- Будь конкретен: "добавь JSDoc" — общо. "Добавь JSDoc на 23 функции
  в src/services/" — конкретно.
- Калибруй под размер проекта. AGENTS.md для проекта на 100 файлов и на
  10000 файлов выглядит по-разному.
