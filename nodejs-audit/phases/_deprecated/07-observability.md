# Фаза 7: Observability

## Твоя роль

Ты — SRE. Твоя задача: оценить готовность приложения к production-эксплуатации
с точки зрения наблюдаемости: можно ли понять что происходит когда что-то
ломается.

## Предварительные требования

- Фазы 0-2 завершены.
- Прочитай `reports/01-recon.md`.

## 4 столпа observability

### Столп 1: Логи

Что проверить:

**Структурированные ли логи?**
```bash
# Используется ли logger или console.log?
grep -rn "console\\.log\\|console\\.error\\|console\\.warn" src/ \
  --include="*.ts" --include="*.tsx" | wc -l

# Есть ли структурированный logger?
grep -rEn "pino|winston|bunyan|@logger" package.json src/ | head -10
```

Цель: JSON-логи с timestamp, level, message, context.
Плохо: console.log с интерполяцией строк.

**Уровни?**
- debug — детали для разработки
- info — важные события (запрос, регистрация)
- warn — что-то странное, но работает
- error — что-то сломалось

**Корреляция запросов?**
- Есть ли request ID, прокидываемый через все слои?
- Можно ли по логам восстановить путь одного запроса?

```bash
grep -rEn "requestId|correlationId|traceId" src/ --include="*.ts" | head -10
```

**Что логируется лишнего?** (PII, секреты)
```bash
# Логируем тела запросов? (часто = логируем пароли)
grep -rEn "log.*req\\.body|log.*password|log.*token" src/ --include="*.ts" | head -10
```

### Столп 2: Метрики

Что проверить:

**RED-метрики?** (Rate, Errors, Duration)
- Считаются ли requests per second?
- Считается ли error rate?
- Считается ли latency (p50, p95, p99)?

```bash
grep -rEn "prom-client|@opentelemetry/sdk-metrics|metrics\\." src/ package.json | head -10
```

**Бизнес-метрики?**
- Регистрации, конверсии, ошибки оплат — считаются?

**Health checks?**
```bash
# Есть ли /health и /ready
grep -rEn "/health|/ready|/healthz|/livez" src/ --include="*.ts" | head -10
```

Цель:
- `/health` — приложение работает
- `/ready` — приложение готово принимать трафик (БД доступна, кеш прогрет)

### Столп 3: Трейсинг

Что проверить:

**OpenTelemetry?**
```bash
grep -rEn "@opentelemetry|opentelemetry" package.json | head -5
```

Если используется — хорошо. Если нет — это пробел.

**Контекст пробрасывается через все слои?**
- HTTP запрос → DB запрос → внешний API — связано трейсом?

### Столп 4: Алерты и мониторинг

(Зависит от инфраструктуры, но можно проверить готовность кода.)

**Sentry / Bugsnag / Rollbar для ошибок?**
```bash
grep -rEn "@sentry|bugsnag|rollbar" package.json | head -5
```

**Captured ли все unhandled errors?**
```bash
grep -rEn "uncaughtException|unhandledRejection" src/ --include="*.ts" | head -10
```

## Готовые промты

### Промт: миграция на структурированный logger
```
В проекте используется console.log в N местах. Замени на pino:

1. Установи pino и pino-pretty (для dev)
2. Создай src/lib/logger.ts с настроенным экземпляром:
   - JSON в production
   - Pretty в development (через NODE_ENV)
   - Уровни: debug, info, warn, error
3. Замени все console.log на logger.info, console.error на logger.error,
   console.warn на logger.warn
4. Файлы для замены:
   - src/X.ts
   - src/Y.ts
   - ...

ВАЖНО:
- Не логируй req.body, req.headers целиком (могут быть пароли/токены)
- Логируй только нужные поля
- Каждый файл = отдельный коммит
- После каждого файла — npm run audit:test
```

### Промт: добавить request ID
```
Добавь middleware, который:
1. Генерирует UUID для каждого запроса (если в заголовке нет X-Request-Id)
2. Сохраняет его в AsyncLocalStorage / context
3. Прокидывает в logger автоматически
4. Возвращает в ответе как заголовок X-Request-Id

Все логи внутри обработчика запроса должны содержать этот ID.
```

## Правила

- Если приложение — простой скрипт без production-планов, observability не
  должна быть 10/10. Калибруй под контекст.
- Не предлагай Datadog за $1000/мес для pet-проекта.
- Базовый уровень для серьёзного проекта: pino + Sentry + /health
  эндпоинт. Это не дорого и закрывает 80% боли.

## Чего не делать

- Не меняй код
- Не настраивай реальные интеграции (Sentry, Datadog) — только готовь к этому

## Готов? Начинай с подсчёта console.log.
