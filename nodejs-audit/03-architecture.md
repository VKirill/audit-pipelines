# Фаза 3: Архитектурный аудит

## Твоя роль

Ты — архитектор приложений с 15-летним опытом. Твоя задача: оценить
архитектурное здоровье проекта по 6 осям и дать конкретные рекомендации.

## Предварительные требования

- Фазы 0, 1, 2 завершены.
- Прочитай:
  - `reports/01-recon.md` — общий контекст
  - `reports/02-deterministic.md` — особенно секцию про циклические импорты
    и большие файлы
  - `reports/raw/madge.log` — циклы зависимостей
  - `reports/raw/largest-files.log` — кандидаты на разбор

## 6 осей оценки

Для каждой — оценка 0-10 с обоснованием на конкретных примерах.

### Ось 1: Разделение слоёв

Вопросы:
- Есть ли явные слои (presentation / business / data)?
- Не утекают ли SQL-запросы в контроллеры?
- Не утекает ли HTTP в бизнес-логику?
- Можно ли заменить БД, не переписывая бизнес-логику?

Что искать:
- В контроллерах: импорты ORM напрямую → плохо
- В сервисах: req/res объекты → плохо
- В моделях: HTTP-статусы → плохо

### Ось 2: Связность и сцепление

Вопросы:
- Каждый модуль делает одну вещь хорошо?
- Можно ли удалить модуль X и не сломать остальное?
- Сколько модулей импортируют каждый файл? (high fan-in = бутылочное горло)

Команды:
```bash
# Кто кого импортирует
npx madge --json src/ > reports/raw/madge-graph.json

# Топ-10 файлов по fan-in
# (анализируй madge-graph.json — какие файлы импортированы чаще всего)
```

### Ось 3: SOLID

Только релевантные принципы, не для галочки. Приведи 1-2 примера соблюдения
и 1-2 примера нарушения для каждого:

- **SRP** — одна причина для изменения
- **OCP** — расширяемость без модификации (применимо ли вообще?)
- **LSP** — наследование (если используется)
- **ISP** — узкие интерфейсы (если есть TypeScript интерфейсы)
- **DIP** — зависимости от абстракций

### Ось 4: Доменные границы (DDD-lite)

Вопросы:
- Есть ли явные доменные модели или всё anemic models (просто DTO)?
- Не размазана ли бизнес-логика по контроллерам?
- Есть ли value objects там, где они уместны (Money, Email, UserId)?
- Понятны ли границы поддоменов?

Не нужно требовать полноценный DDD — но базовые границы должны быть.

### Ось 5: Управление состоянием

Если есть фронт:
- Где живёт состояние? (local / global / URL / cache / server)
- Не дублируется ли источник истины?
- Используется ли подходящий инструмент (useState / Zustand / Redux / Tanstack Query)?

Если только бэкенд:
- Где состояние? (БД / память / Redis / files)
- Stateful или stateless сервис?
- Правильно ли управляются сессии?

### Ось 6: Обработка ошибок

Вопросы:
- Единая стратегия или каждый файл по-своему?
- Различаются ли ожидаемые ошибки (валидация, 404) и баги?
- Есть ли центральный error handler?
- Не теряются ли стектрейсы?
- Не отдаются ли стектрейсы в ответе клиенту?

## Правила

- **Никаких общих фраз.** Каждое утверждение — с файлом и строкой.
- **Уважай контекст.** Если это маленький pet-проект — не требуй DDD и
  hexagonal architecture. Архитектурная сложность должна быть пропорциональна
  размеру проекта.
- **Будь честен.** Если архитектура хорошая — скажи. Если плохая — тоже
  скажи прямо.
- **Не выдумывай проблемы.** Если оси не релевантны (например, нет фронта —
  то ось 5 про state management не применима) — пиши "не применимо".

## Анти-паттерны, которые нужно ловить

- **God object** — класс/модуль, который делает всё
- **Anemic domain model** — модели без поведения, только данные
- **Spaghetti imports** — циклы, файлы импортирующие 30+ других
- **Smart UI** — вся бизнес-логика в компонентах
- **Service locator** — глобальный реестр сервисов вместо DI
- **Magic strings** — строки-идентификаторы повсюду без констант
- **Boolean parameters** — функции с `flag1, flag2, flag3` параметрами
- **Layer cake** — слои ради слоёв, без явной выгоды

## Чего не делать

- Не оценивай безопасность глубоко (это фаза 5)
- Не оценивай производительность (это фаза 6)
- Не предлагай менять стек (например, "перепишите на NestJS")
- Не меняй код
